# Quant Lab Export Storage Report Extension

> Date: 2026-05-05 KST
> Scope: product-depth extension after `QUANT_LAB_EXPORT_CLEANUP_PREVIEW_EXTENSION.md`
> Status: implemented and verified in the local environment

## Executive Summary

The Fincept-compatible Quant Lab contract remains complete. This run closed the next practical export-operability gap: operators could generate, verify, list, and clean up exports for one saved run, but they could not see cross-run export storage pressure before choosing what to inspect.

The new workflow is read-only:

```text
all saved Quant Lab runs -> scan generated exports -> storage report -> operator review
```

It does not delete files, recompute backtests, rewrite manifests, or touch Qlib provider output.

## What Changed

### Cross-Run Storage Scanner

Added:

- `summarize_backtest_artifact_export_storage(...)`
- `export_storage_report(...)`
- `GET /api/v1/quant/exports/storage`

The scanner walks `data/quant_lab/backtests/*/exports/*` and reports:

- total saved run directories
- runs with generated exports
- total export directories
- total generated bytes
- total exported rows
- format counts
- export manifest status counts
- oldest and newest export timestamps
- top storage-heavy runs
- old export candidates by age

### Fault Tolerance

The report tolerates broken export state:

- missing `exports/` directory
- run directory with no exports
- export directory with missing `export_manifest.json`
- corrupt export manifest JSON
- partial integrity metadata

Those cases are surfaced through `manifest_status_counts` and per-run status fields rather than failing the entire report.

### API Surface

```http
GET /api/v1/quant/exports/storage?limit=20&stale_after_days=30
```

Example response shape:

```json
{
  "status": "success",
  "schema_version": "quant_lab_export_storage_report_v1",
  "run_count": 12,
  "runs_with_exports": 5,
  "export_directory_count": 18,
  "total_bytes": 1243881,
  "total_rows": 912,
  "format_counts": {"csv": 7, "jsonl": 8, "parquet": 3},
  "manifest_status_counts": {"success": 18},
  "top_runs": [],
  "stale_exports": []
}
```

### UI Workflow

The Quant Lab Run History panel now has a `storage report` action. It renders:

- cross-run summary chips
- format and manifest status rows
- artifact root
- largest runs by generated export storage
- old export candidates
- open buttons for large runs

The UI remains review-oriented. Cleanup actions still live in the per-run preview/apply workflow.

- No export cleanup during report generation.

## Practical Improvement Points

### 1. Add Offline Export Verification CLI

Next best increment:

```powershell
python scripts\verify_quant_export.py data\quant_lab\backtests\{run_id}\exports\{export_id}
```

It should load `export_manifest.json`, re-hash files, compare SHA-256 and byte sizes, emit JSON plus human-readable output, and exit non-zero on drift.

### 2. Add Export Package Manifest

Create `package_manifest.json` per export with:

- package schema version
- source run id
- source artifact manifest hash
- source config hash
- export manifest hash
- file hashes
- generated-by route or CLI
- created-at timestamp

This prepares the system for signed export packages.

### 3. Add Cross-Run Cleanup Preview

Only after operators trust the read-only report, add:

```http
POST /api/v1/quant/exports/cleanup-preview
```

Keep apply separate and require an exact preview id or exact candidate list.

### 4. Add Replay Report Drilldown

Add read-only side-by-side replay report comparison:

- report A
- report B
- original metrics
- replay metrics
- delta differences
- tolerance policy differences
- code version differences
- data snapshot differences

This should not recompute a backtest.

### 5. Add Storage Trend Artifacts

Persist daily storage snapshots under `reports/quant_lab_export_storage/` so automation can show storage trends without rescanning when the run tree grows.

## Final Assessment

The two source MD files now reflect the current implementation state. The compatibility contract is still complete, and the export subsystem now has a safer operator path:

```text
generate -> verify -> list -> cleanup preview/apply per run -> cross-run storage report
```

The most practical next slice is offline export-package verification, because it extends the existing integrity metadata to copied artifacts without requiring a running API server.
